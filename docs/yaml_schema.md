# YAML 剧本 Schema 设计文档

## 设计目标

这个 Schema 服务于一个明确的场景：**小说作者将 .txt 稿件交给 LLM，快速获得一份可编辑、可打磨的剧本初稿**。

因此每个设计决策都在三个目标之间权衡：

| 目标 | 说明 |
|------|------|
| **LLM 友好** | 结构足够简单，模型能稳定输出合法 YAML，不产生幻觉字段 |
| **作者友好** | 作者拿到 YAML 后能直观理解、直接修改，不需要学新工具 |
| **剧本专业性** | 遵循影视编剧的基本格式惯例，不丢失小说的叙事节奏 |

---

## Schema 总览

```yaml
title: "剧本标题"
scenes:              # 场次列表
  - scene_number: 1
    slugline: "内. 古宅门厅 - 夜"
    content:         # 内容序列（保持动作与对白的交错顺序）
      - type: "action"
        text: "林侦探推开吱呀作响的橡木门，走进门厅。"
      - type: "dialogue"
        character: "林侦探"
        parenthetical: "警惕地环顾四周"
        line: "有人在吗？"
      - type: "action"
        text: "寂静。只有风声穿过破窗的缝隙。"
characters:          # 角色清单
  - name: "林侦探"
    description: "一名谨慎冷静的侦探，对案件执着"
```

---

## 字段详解与设计原因

### 1. `title` — 剧本标题

```
类型: string
必填: 否（默认 "Untitled"）
```

**设计原因**：最基础的元素。设为非必填是因为 LLM 有时无法从片段中推断出合适标题，此时留空比编一个离谱的标题更好。作者拿到初稿后自己填。

---

### 2. `scenes` — 场次列表

```
类型: Scene 数组
必填: 否
```

**设计原因**：场（Scene）是剧本的基本单元。一部剧本由若干场戏组成，每场戏发生在一个特定的时间和地点。选择以 Scene 而非 Act（幕）作为顶层分组，原因是：

- **LLM 切分可靠性**：模型识别"地点变了"或"时间跳了"比识别"这是第二幕开端"可靠得多。幕是高级叙事概念，模型容易分错，作者自己标注更准。
- **编辑粒度**：作者修改时通常以"场"为单位——某场戏重写、某场戏删掉、两场对调顺序。以场为顶层，直接在 `scenes` 列表里增删改查即可。
- **幕可以后加**：如果作者需要三幕结构，可以在打磨阶段用注释或额外字段标注，不阻塞初稿生成。

---

### 3. `scene_number` — 场次序号

```
类型: int
必填: 是
```

**设计原因**：编号让作者能快速定位（"第 3 场需要重写"）。选择数字而非字符串是为了排序可靠——数字不会出现 "10" 排在 "2" 前面的问题。

---

### 4. `slugline` — 场标

```
类型: string
必填: 否（默认 ""）
格式: "内/外. 地点 - 时间"
示例: "内. 侦探办公室 - 黄昏"
```

**设计原因**：场标（Slugline / Scene Heading）是好莱坞剧本的通用格式。三个要素：

- **内/外（INT./EXT.）**：决定灯光、布景方式，是拍摄的核心信息
- **地点**：场景发生的具体位置
- **时间**：一天中的时段（晨/昼/黄昏/夜），决定光影基调

为什么放在一个字符串里而不是拆成三个字段？**为了让 LLM 更容易输出正确格式**。多字段意味着模型可能在 `location` 里写时间、在 `time` 里写地点。一个字段 + 格式约定 = 更少的解析错误。作者编辑时也只需改一个字符串，符合直觉。

---

### 5. `content` — 内容序列（核心设计）

```
类型: ContentItem 数组
必填: 否
```

**这是整个 Schema 最关键的设计决策。**

#### 问题：为什么不用"对白列表 + 舞台指示"的两段式结构？

小说叙事中，动作描写和对白是**交错**的：

> 他推开门（动作），"有人在吗？"（对白），无人应答（动作），"我知道你在里面。"（对白）

如果用两段式结构——

```yaml
# 反例：丢失顺序
dialogues:
  - character: "他"
    line: "有人在吗？"
  - character: "他"
    line: "我知道你在里面。"
stage_directions: "他推开门。无人应答。"
```

——作者拿到这份 YAML 后**无法知道哪句对白对应哪个动作**，需要回原文查找，失去了"快速可编辑"的意义。

#### 解决方案：`content` 序列

```yaml
content:
  - type: "action"
    text: "他推开门。"
  - type: "dialogue"
    character: "他"
    line: "有人在吗？"
  - type: "action"
    text: "无人应答。"
  - type: "dialogue"
    character: "他"
    line: "我知道你在里面。"
```

**顺序即是节奏**。作者读到 content 列表，就能按序理解整场戏的演出流程。修改时也直观——在列表中插入、删除、调序即可。

`type` 字段用字符串枚举（`"action"` / `"dialogue"`）而非数字，因为 YAML 阅读者不需要记住 0 和 1 分别代表什么。

---

### 6. `type: "action"` — 动作/舞台指示

```yaml
- type: "action"
  text: "动作或舞台指示的描述文字"
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | string | 固定值 `"action"` |
| `text` | string | 动作/场景的描述 |

**设计原因**：对应剧本格式中的 Action Lines。描述角色的动作、场景的视觉变化、氛围等。`text` 只用单个字符串而非分割成多个段落，原因是 LLM 自己控制分段更自然，而且作者在 YAML 编辑器中修改多行字符串（用 `|` 或 `>` 语法）也很方便。

---

### 7. `type: "dialogue"` — 对白

```yaml
- type: "dialogue"
  character: "林侦探"
  parenthetical: "警惕地环顾四周"
  line: "有人在吗？"
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `character` | string | 是 | 说话的角色名 |
| `line` | string | 是 | 台词正文 |
| `parenthetical` | string | 否 | 情绪/动作提示（对应剧本格式中的括号注） |

**设计原因**：

- **`character` 和 `characters` 顶层列表的关系**：`character` 是引用字符串，`characters` 是角色定义表。用字符串引用而非结构化嵌套，是因为同一角色会在多处说话，复制整个角色对象会造成冗余和不一致。
- **`parenthetical`（括号注）**：标准剧本格式中，括号注出现在角色名和台词之间，用圆括号括起，提示演员这句话的表演方式。例如 `(冷笑)`、`(压低声音)`。独立字段让作者可以快速扫描所有情绪提示——如果和台词揉在一起就难以检索。
- **为什么 `character` 和 `line` 是必填**：没有角色名的对白没有意义；没有台词的对白不是对白（那是动作）。

---

### 8. `characters` — 角色清单

```yaml
characters:
  - name: "林侦探"
    description: "一名谨慎冷静的侦探，对案件执着"
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `name` | string | 是 | 角色名 |
| `description` | string | 否 | 角色简介 |

**设计原因**：

- **集中管理**：所有角色在一个地方定义，作者可以快速浏览整个角色表。后续扩展时自然加入更多属性（如年龄、外貌、人物弧光）。
- **`name` 必填、`description` 非必填**：LLM 有时只能从片段中抽取角色名但无法给出详细描述。允许留空比强制模型编造要好。
- **为什么是顶层字段而非每场戏内嵌**：一个角色贯穿多场戏，角色表应该全局唯一。各场戏通过 `content[].character` 的字符串引用角色名，不需要重复定义。

---

## 为什么不包含的元素

| 未包含 | 原因 |
|--------|------|
| **Act（幕）** | 幕是分析工具而非生成工具。LLM 分幕不准确，作者打磨阶段自己标注更合理 |
| **Transition（转场）** | CUT TO、FADE IN 等转场指令属于后期润色，初稿阶段增加噪音 |
| **Scene 级别的 characters_present** | 与 content 序列中的角色引用重复。从 content 中可以自动提取每场出现的角色 |
| **页码/时长估算** | 属于排版工具的功能，不属于剧本 Schema |

## 与标准剧本格式的对应关系

```
标准剧本格式                    YAML Schema
─────────────────────────────────────────────
INT. OFFICE - DUSK      →    slugline: "内. 办公室 - 黄昏"

DETECTIVE LIN           →    - type: "dialogue"
(gravely)                       character: "林侦探"
I've seen this before.          parenthetical: "阴沉地"
                                line: "我见过这种情况。"

Lin opens the drawer.   →    - type: "action"
A single bullet rolls          text: "林打开抽屉。一颗子弹滚了出来。"
out.

CHARACTER LIST          →    characters:
  DETECTIVE LIN ...               - name: "林侦探"
                                    description: "..."
```

## 扩展预留

当前 Schema 是初版最小集，以下扩展点已在结构中预留：

1. **`characters` 扩展** — 给 `CharacterInfo` 加字段（age, gender, appearance, arc）不影响现有解析
2. **`content` 新增类型** — 可在 `type` 枚举中加入 `"transition"`、`"shot"`、`"note"` 等
3. **Scene 元数据** — 每场戏可加 `estimated_duration`、`tone`、`conflict_type` 等
4. **多幕结构** — 顶层可加 `acts` 字段，将 `scenes` 按幕分组，向后兼容
